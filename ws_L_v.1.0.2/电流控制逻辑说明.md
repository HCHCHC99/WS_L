# 电流控制逻辑说明

## 一、整体架构（三层结构）

整个电流控制逻辑分为三层，通过 EventBus 事件总线通信：

```
dev_sensor (电流检测层)
    |
    |-- EventBus 发布 TOPIC_CURRENT_ALARM
    |
    +---> dev_rturn (转向锁死层)  -- 通过 RTurn_OnCurrentAlarm 回调接收
    |
    +---> dev_motor (电机仲裁层)  -- 通过 Motor_OnCurrentAlarm 回调接收
    |
    +---> App_FaultHandler (故障管理层) -- 通过 FaultHandler_OnCurrentAlarm 回调接收
```

---

## 二、dev_sensor -- 电流检测层

### 2.1 电流计算

传感器输出特性：VOUT = 2500 + IP x 400 (mV)

其中：
- IP 为电流(A)
- VOUT 为传感器输出电压(mV)

反推电流公式：
```
电流(mA) = (VOUT - 2500) x 1000 / 400
```

例如：
- VOUT = 2500mV -> 电流 = 0mA (零点)
- VOUT = 2900mV -> 电流 = 1000mA (1A)
- VOUT = 3300mV -> 电流 = 2000mA (2A)

### 2.2 过流检测逻辑

过流检测使用绝对值比较，正反向电流都检测：

```
触发阈值 = |Threshold| + |Hysteresis|
解除阈值 = |Threshold| - |Hysteresis|
```

例如：Threshold=2000mA, Hysteresis=200mA
- 触发阈值 = 2200mA (电流超过2200mA触发)
- 解除阈值 = 1800mA (电流降到1800mA以下解除)

### 2.3 窗口防抖（两种模式可选）

为了防止电流瞬间波动导致误触发，有两种防抖模式：

#### 模式A：点数模式 (OVERCURRENT_MODE_SAMPLE_COUNT)

- 触发：连续 N 次采样都过流，才真正触发告警
- 解除：连续 M 次采样都正常，才真正解除告警
- N 和 M 通过配置参数 TriggerWindowSize 和 ReleaseWindowSize 设置

例如：TriggerWindowSize=3，需要连续3次采样都过流才触发

#### 模式B：时间模式 (OVERCURRENT_MODE_TIME_WINDOW)

- 触发：持续过流超过 T ms，才真正触发告警
- 解除：持续正常超过 T ms，才真正解除告警
- T 通过配置参数 TriggerWindowMs 和 ReleaseWindowMs 设置

例如：TriggerWindowMs=100，需要持续过流100ms才触发

### 2.4 告警发布

触发时发布：
```
EventBus_Publish(TOPIC_CURRENT_ALARM, &stcEvent)
其中 stcEvent.u8IsActive = 1
```

解除时有两种模式：

#### 自动清除模式 (OVERCURRENT_CLEAR_AUTO)

电流恢复正常后，窗口防抖通过，自动发布解除事件：
```
EventBus_Publish(TOPIC_CURRENT_ALARM, &stcEvent)
其中 stcEvent.u8IsActive = 0
```

#### 手动清除模式 (OVERCURRENT_CLEAR_MANUAL)

电流恢复正常后，不发布解除事件。告警状态保持，电机 block 保持。
需要通过 Modbus 写 REG_FAULT_STATUS 手动清除。

## 三、dev_motor -- 电机仲裁层

### 3.1 过流触发后的行为

当收到 TOPIC_CURRENT_ALARM 事件 (u8IsActive=1)：

1. Motor_OnCurrentAlarm 回调被调用
2. 向 block_fwd 队列添加 DEV_ID_OVERCUR_FWD 阻塞指令
3. 仲裁器重新仲裁，正转被阻塞，电机停止

注意：只阻塞正转方向，不阻塞反转方向。

### 3.2 为什么只阻塞正转？

- 反转（关闭）时过流属于预期工况（机械限位），由 dev_rturn 的角度限位处理
- 正转（打开）时过流才是真正的异常（堵转、卡死）

### 3.3 过流解除后的行为

当收到 TOPIC_CURRENT_ALARM 事件 (u8IsActive=0)：

1. Motor_OnCurrentAlarm 回调被调用
2. 从 block_fwd 队列移除 DEV_ID_OVERCUR_FWD
3. 仲裁器重新仲裁，正转恢复可用

### 3.4 手动清除接口

Motor_ClearOvercurrentBlock() 函数：
- 直接从 block_fwd 移除 DEV_ID_OVERCUR_FWD
- 作为双重保险，在 FaultHandler_ClearFault 中调用

## 四、dev_rturn -- 转向锁死层

### 4.1 过流触发后的行为

当收到 TOPIC_CURRENT_ALARM 事件 (u8IsActive=1)：

1. RTurn_OnCurrentAlarm 回调被调用
2. 遍历所有设备，找到 RTurn 设备
3. 调用 RTurn_HandleOvercurrent 处理过流

### 4.2 RTurn_HandleOvercurrent 处理逻辑

读取电机仲裁器的期望方向（不是实际物理方向），根据期望方向处理：

#### 情况A：期望方向为正转（打开）

- 已校准状态：锁死正转方向，不重置角度
- 未校准状态：锁死正转方向，不校准、不重置角度

#### 情况B：期望方向为反转（关闭）

- 已校准状态：锁死反转方向，重置角度为下限位角度 (fMinAngle)
- 未校准状态：锁死反转方向，校准成功 (u8Calibrated=1)，重置角度为下限位角度

### 4.3 锁死后的影响

锁死后发布 TOPIC_RTURN_LIMIT 事件：
- Motor_OnRTurnLimit 收到后向 block_fwd 或 block_rev 添加阻塞指令
- 电机仲裁器重新仲裁，对应方向被阻塞

### 4.4 主动轮询机制

在 RTurn_UpdateAngle 中，每次更新都主动检查：

```
如果：没有锁死 AND 期望方向有效 AND 限位未触发
那么：主动读取传感器告警状态
  如果告警：立即锁死（不依赖过流事件）
```

为什么需要主动轮询？
- 持续过流时，dev_sensor 不会重复发布事件（因为告警状态已经为1）
- 但仲裁器可能先输出 STOP，然后仲裁器又输出方向
- 此时需要主动检测告警状态，确保锁死

### 4.5 锁死解除条件

锁死解除条件：期望方向变为相反方向

例如：
- 正转锁死中，期望方向变为反转 -> 解除正转锁死
- 反转锁死中，期望方向变为正转 -> 解除反转锁死

解除后立即重新检测告警状态：
- 如果仍然过流，锁死新方向
- 如果正常，不锁死

### 4.6 角度积分在锁死时的行为

- 反转锁死：跳过角度积分，角度冻结在 fMinAngle
- 正转锁死：允许继续积分（霍尔计数继续累加），直到电机停止

## 五、App_FaultHandler -- 故障管理层

### 5.1 故障位定义

```
FAULT_BIT_OVERVOLTAGE  = bit0  (0x0001)  -- 过压故障
FAULT_BIT_UNDERVOLTAGE = bit6  (0x0040)  -- 欠压故障
FAULT_BIT_OVERCURRENT  = bit2  (0x0004)  -- 过流故障
```

### 5.2 过流故障设置

当收到 TOPIC_CURRENT_ALARM 事件 (u8IsActive=1)：
```
RealTime_SetFault(FAULT_BIT_OVERCURRENT)
```

### 5.3 过流故障清除

当收到 TOPIC_CURRENT_ALARM 事件 (u8IsActive=0)：
```
RealTime_ClearFault(FAULT_BIT_OVERCURRENT)
```

### 5.4 手动清除流程

通过 Modbus 写 REG_FAULT_STATUS (0x2722) 触发 FaultHandler_ClearFault：

```
写 REG_FAULT_STATUS (bit2=1)
  -> FaultHandler_ClearFault(FAULT_TYPE_OVERCURRENT)
     |
     +-> 1. Sensor_Device_ClearAlarm()
     |     清除传感器告警状态
     |     发布 TOPIC_CURRENT_ALARM (u8IsActive=0)
     |     -> Motor_OnCurrentAlarm 移除 block_fwd 的 DEV_ID_OVERCUR_FWD
     |
     +-> 2. Motor_ClearOvercurrentBlock()
     |     直接移除 block_fwd 的 DEV_ID_OVERCUR_FWD（双重保险）
     |
     +-> 3. RealTime_ClearFault(FAULT_BIT_OVERCURRENT)
           清除故障寄存器
```

## 六、完整过流场景流程

### 6.1 过流触发流程

```
电机正转 -> 堵转 -> 电流上升 -> ADC检测到过流
  |
  +-> dev_sensor: 窗口防抖通过
  |   -> 发布 TOPIC_CURRENT_ALARM (u8IsActive=1)
  |
  +-> dev_motor: Motor_OnCurrentAlarm
  |   -> block_fwd 添加 DEV_ID_OVERCUR_FWD
  |   -> 仲裁器重新仲裁 -> 正转被阻塞 -> 电机停止
  |
  +-> dev_rturn: RTurn_OnCurrentAlarm
  |   -> RTurn_HandleOvercurrent
  |   -> 锁死正转方向
  |   -> 发布 TOPIC_RTURN_LIMIT
  |   -> Motor_OnRTurnLimit -> block_fwd 添加 DEV_ID_RTURN_FWD（双重锁死）
  |
  +-> App_FaultHandler: FaultHandler_OnCurrentAlarm
      -> 设置 FAULT_BIT_OVERCURRENT
```

### 6.2 过流恢复流程（手动清除模式）

```
Modbus 写 REG_FAULT_STATUS (bit2=1)
  -> FaultHandler_ClearFault(FAULT_TYPE_OVERCURRENT)
     |
     +-> Sensor_Device_ClearAlarm()
     |   -> 发布 TOPIC_CURRENT_ALARM (u8IsActive=0)
     |   -> Motor_OnCurrentAlarm -> 移除 block_fwd 的 DEV_ID_OVERCUR_FWD
     |
     +-> Motor_ClearOvercurrentBlock()
     |   -> 直接移除 block_fwd 的 DEV_ID_OVERCUR_FWD（双重保险）
     |
     +-> RealTime_ClearFault(FAULT_BIT_OVERCURRENT)
         -> 清除故障寄存器
```

### 6.3 过流恢复流程（自动清除模式）

```
电流恢复正常 -> 窗口防抖通过
  -> dev_sensor 发布 TOPIC_CURRENT_ALARM (u8IsActive=0)
     |
     +-> Motor_OnCurrentAlarm -> 移除 block_fwd 的 DEV_ID_OVERCUR_FWD
     |
     +-> FaultHandler_OnCurrentAlarm -> 清除 FAULT_BIT_OVERCURRENT
```

## 七、关键设计要点

### 7.1 双重锁死机制

过流触发时，dev_motor 和 dev_rturn 都会锁死正转方向：
- dev_motor：通过 DEV_ID_OVERCUR_FWD 阻塞 block_fwd
- dev_rturn：通过 DEV_ID_RTURN_FWD 阻塞 block_fwd（通过 TOPIC_RTURN_LIMIT 事件）

这样即使一个锁死被意外清除，另一个仍然有效。

### 7.2 主动轮询 vs 事件驱动

dev_rturn 同时使用两种方式检测过流：
- 事件驱动：通过 TOPIC_CURRENT_ALARM 回调
- 主动轮询：每次 Update 时主动读取传感器告警状态

这是因为持续过流时不会重复发布事件，但仲裁器状态可能变化。

### 7.3 正转 vs 反转过流处理差异

| 方向 | dev_motor | dev_rturn |
|------|-----------|-----------|
| 正转过流 | 阻塞正转 | 锁死正转，不重置角度 |
| 反转过流 | 不处理 | 锁死反转，重置角度为下限位 |

反转过流被视为正常工况（机械限位），所以 dev_motor 不处理。

### 7.4 手动清除 vs 自动清除

| 模式 | 电流恢复后行为 | 适用场景 |
|------|---------------|---------|
| 自动清除 | 自动解除告警，自动移除 block | 需要自动恢复的场景 |
| 手动清除 | 告警保持，block 保持，需手动清除 | 需要人工确认的场景 |
