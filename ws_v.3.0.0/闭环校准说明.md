# BLDC Hall Sensor Calibration: Principle and Process

## Overview

The BLDC motor controller uses 6-step (trapezoidal) commutation with 3-channel Hall-effect sensors for rotor position feedback. The relationship between Hall sensor states and commutation steps depends on:

1. **Motor physical construction** — magnet pole placement, winding layout
2. **Hall sensor mounting orientation** — mechanical offset relative to stator windings
3. **Phase wiring** — how the 3 motor phases connect to the 3 inverter outputs
4. **Hall wiring** — how the 3 Hall sensors connect to the 3 GPIO inputs

Any change to these connections invalidates the Hall-to-Step lookup table, requiring re-calibration.

---

## Background: 0-Degree vs 90-Degree Offset

### The Two Angles

In a 6-step BLDC drive, each electrical revolution is divided into 6 sectors of 60 degrees. Two different "angles" describe the stator-rotor relationship:

| Angle | Meaning | Purpose |
|-------|---------|---------|
| **0° offset** | Stator field **aligned with** rotor field | Zero torque; rotor follows stator in open-loop |
| **90° offset** | Stator field **leads** rotor field by 90° | Maximum torque (sin 90° = 1.0) for closed-loop |

### Why Calibration Captures 0-Degree Offset

During open-loop operation at constant speed with no load:

- The stator field rotates at a known rate (e.g., step 0→1→2→3→4→5→0...)
- The rotor **follows** the stator field with a small mechanical lag (typically 5–15 degrees)
- At each Hall edge (rotor sector boundary), we record: **"Hall state X → stator is at step Y"**

This directly yields the **0-degree offset mapping**. From this, the 90-degree offset tables for CW and CCW closed-loop operation are derived by adding a fixed step offset:

```
CW  (stator leads rotor by 90° CW):   hall2step[H] = (calib[H] + offset_cw)  % 6
CCW (stator leads rotor by 90° CCW):  hall2step[H] = (calib[H] + offset_ccw) % 6
```

For this specific motor, the experimentally determined offsets are:

- **CW offset = +5** (equivalent to -1, i.e., stator retarded by 60° CCW)
- **CCW offset = +2**

These are **not** symmetric (±1) because of the motor's Hall sensor mounting orientation. The calibration process automatically accounts for this.

## Calibration Principle

### Event-Driven Edge Detection

Instead of periodic GPIO sampling and majority voting, the calibration uses **event-driven Hall edge detection**. The principle is:

```
For each Hall state change (rising or falling edge on any of the 3 Hall inputs):
    1. Read the 3-bit Hall code (0x01 through 0x06)
    2. Read the current commutation step (s_comm_step, 0 through 5)
    3. Record: hall2step[hall_state] = s_comm_step
```

After observing all 6 valid Hall states, the mapping is validated:

1. **Bijection check**: each Hall state (0x01–0x06) maps to exactly one step (0–5)
2. **Consistency check**: the mapping matches across multiple electrical revolutions
3. **Invalid state rejection**: Hall codes 0x00 (000) and 0x07 (111) are inherently invalid

### Why This Works

During open-loop rotation at constant speed:

- The stator field rotates in fixed 60-degree steps (0→1→2→3→4→5→0...)
- The rotor follows the stator field closely (small lag under no load)
- Hall edges fire when the rotor crosses sector boundaries
- At each edge, the stator is at a specific commutation step

The recorded mapping **directly** captures: "when the rotor is at Hall position H, energize step S to align the stator with the rotor." This is the 0-degree offset table.

## Calibration Process (comm_mode = 5)

### Step-by-Step Flow

```
1. Set comm_mode = 5 in Keil Watch

2. Motor enters calibration:
   ├── Calibration globals are reset
   ├── Hall FSM is stopped (ISRs still fire for edge timing)
   ├── Open-loop rotation begins at constant interval (5000 us/step)
   └── 500 ms settling period (motor accelerates to steady speed)

3. After settling, edge detection begins:
   ├── Every ~1 ms: hall_3ch_read_raw() reads GPIO state
   ├── When Hall state changes → edge detected
   │   ├── Record: pending_map[hall] = s_comm_step
   │   └── Log: [CAL] edge: hall=0x0X step=Y
   ├── When all 6 Hall states observed:
   │   ├── Validate bijection (one-to-one mapping)
   │   ├── Validate consistency with reference cycle
   │   ├── If valid: cycle_count++, lock reference table
   │   └── If invalid: cycle_rejected++
   └── After 12 valid cycles → SUCCESS
   
4. On success:
   ├── g_calib_status = CALIB_SUCCESS (2)
   ├── g_calib_table[1..6] = 0-degree offset mapping
   └── Motor stops automatically

5. On failure (3-second timeout):
   ├── g_calib_status = CALIB_FAIL_TIMEOUT (3)
   ├── g_calib_error.hall_state = bitmask of unseen Hall states
   └── Check g_calib_hall_seen[1..6] and g_calib_hall_obs[1..6]
```

### Key Timings

| Parameter | Value | Purpose |
|-----------|-------|---------|
| Step interval | 5000 us | Matches OPEN_FW default speed |
| Settling delay | 500 ms | Motor reaches steady open-loop speed |
| Timeout | 3000 ms | Overall calibration timeout |
| Target cycles | 12 | Valid electrical revolutions for confidence |
| Main loop rate | ~1 kHz | Approximate, varies with Modbus polling |

## Derived Closed-Loop Modes

### comm_mode = 6 — Calibrated CW Closed-Loop

```
Process:
  1. Build CW table: hall2step[H] = (g_calib_table[H] + 5) % 6
  2. Open-loop ramp using fly-start parameters (ol_fly_start -> ol_fly_target)
  3. After ramp → flying start into closed-loop using derived CW table
  4. Hall ISR drives commutation via the derived table
```

### comm_mode = 7 — Calibrated CCW Closed-Loop

```
Process:
  1. Build CCW table: hall2step[H] = (g_calib_table[H] + 2) % 6
  2. Open-loop ramp using fly-start parameters
  3. After ramp → flying start into closed-loop using derived CCW table
  4. Hall ISR drives commutation via the derived table
```

## Keil Watch Variables

### During Calibration (comm_mode = 5)

| Variable | Type | Meaning |
|----------|------|---------|
| `g_calib_status` | enum | 0=IDLE, 1=RUNNING, 2=SUCCESS, 3=TIMEOUT, 4=STALL, 5=MISSING, 6=DUPLICATE, 7=INVALID, 8=AMBIGUOUS |
| `g_calib_valid_cycles` | uint8 | Completed valid cycles (target: 12) |
| `g_calib_total_steps` | uint8 | Total step transitions (confirms motor spinning) |
| `g_calib_cycle_total` | uint8 | Total cycles attempted |
| `g_calib_cycle_rejected` | uint8 | Cycles that failed bijection or consistency |
| `g_calib_hall_seen[8]` | uint8[8] | 1 = Hall state observed at least once; [0] and [7] should be 0 |
| `g_calib_hall_obs[8]` | uint8[8] | Total observation count per Hall state |
| `g_calib_hall_first_step[8]` | uint8[8] | First step each Hall state appeared on (0–5) |
| `g_calib_hall_first_cycle[8]` | uint8[8] | First cycle each Hall state appeared in (1-based) |
| `g_calib_error` | struct | Error detail (hall_state bitmask, step_a, step_b, reserved) |

### After Successful Calibration

| Variable | Meaning |
|----------|---------|
| `g_calib_table[1..6]` | **0-degree offset mapping**: Hall state → commutation step |

### During Derived Closed-Loop (comm_mode = 6 or 7)

| Variable | Meaning |
|----------|---------|
| `g_calib_status` | Should be 2 (SUCCESS) from prior calibration |
| `g_calib_table[1..6]` | Source 0-degree table used for derivation |

## Calibration Status Codes

| Code | Name | Trigger | Diagnosis |
|------|------|---------|-----------|
| 0 | IDLE | Not running | — |
| 1 | RUNNING | Calibration in progress | Watch `g_calib_total_steps` to verify motor spinning |
| 2 | SUCCESS | 12 valid cycles completed | `g_calib_table` is valid |
| 3 | TIMEOUT | 3 seconds elapsed | Check `g_calib_hall_seen[1..6]` — which are 0? |
| 4 | STALL | Hall states stopped changing | Motor not spinning, or Hall sensors disconnected |
| 5 | MISSING | Some Hall state never observed | Check `g_calib_error.hall_state` for the missing code |
| 6 | DUPLICATE | Hall state mapped to multiple steps | Check `g_calib_error.step_a` and `step_b` |
| 7 | INVALID | Detected 0x00 or 0x07 repeatedly | Wiring issue or noise |
| 8 | AMBIGUOUS | Majority vote below threshold | (Legacy — should not occur in event-driven mode) |

## Diagnosing Hall Sensor Faults

### Using RTT Logs

During calibration (comm_mode = 5), each Hall edge is logged:

```
[CAL] edge: hall=0x02 step=4 (total_steps=98)
[CAL] edge: hall=0x05 step=1 (total_steps=99)
[CAL] edge: hall=0x03 step=5 (total_steps=100)
...
```

A healthy calibration shows all 6 Hall codes (0x01–0x06) appearing in a consistent, repeating sequence.

### Using Keil Watch

After a TIMEOUT failure (status = 3):

1. **Check `g_calib_hall_seen[1..6]`**: Any entry = 0 means that Hall state was **never observed**. This indicates:
   - A faulty Hall sensor
   - A broken wire
   - A magnet pole defect on the rotor

2. **Check `g_calib_hall_obs[1..6]`**: Large discrepancies in observation counts suggest intermittent sensor behavior.

3. **Check `g_calib_error.hall_state`**: After timeout, this is a **bitmask** of unseen Hall states (bit 1 = Hall 0x01, bit 2 = Hall 0x02, ...). A value of `0x7E` (bits 1–6 all set) means NO Hall states were ever observed — check all wiring.

### Expected Hall Sequence (This Motor)

From the Hall ISR logs (IDLE state during open-loop):

```
CW rotation: 0x05 → 0x04 → 0x06 → 0x02 → 0x03 → 0x01 → (repeat)
CCW rotation: 0x01 → 0x03 → 0x02 → 0x06 → 0x04 → 0x05 → (repeat)
```

## Result Validation

### Calibration Table vs. Manual Table

The manually measured tables for this motor:

```
s_hall2step_cw  = {-, 5, 3, 4, 1, 0, 2, -}  (90° offset, CW closed-loop)
s_hall2step_ccw = {-, 2, 0, 1, 4, 3, 5, -}  (90° offset, CCW closed-loop)
```

From calibration (example output):

```
g_calib_table = {-, 0, 4, 5, 2, 1, 3, -}  (0° offset)
```

Validation:

```
CW derived:  (calib + 5) % 6 = {-, 5, 3, 4, 1, 0, 2, -} == s_hall2step_cw  ✓
CCW derived: (calib + 2) % 6 = {-, 2, 0, 1, 4, 3, 5, -} == s_hall2step_ccw ✓
```

The calibration-derived tables **exactly match** the manually measured tables, confirming the method is correct.
